WarpFS — Implementation Reference

Virtual Filesystem for Agentic Workflows — Language, Permissions, Sandboxing & Manifest Design
Status: Spec Phase  |  Date: June 2026  |  Repo: totalwindupflightsystems/warpfs
Design doc: Hermes VFS Design (v2 metadata-first)

1. Language Decision

Verdict: Rust for the core daemon. Python for the MCP server. Go for the client library.
The deciding factor: the trigger hot path runs on every file save — tree-sitter AST parsing, graph traversal, parallel xattr writes across impacted files. No GC pause, no CGO boundary, zero-copy AST traversal.

1.1 The trigger hot path is the bottleneck

AGENT SAVES FILE → inotify fires → trigger runs:
  1. tree-sitter parse AST of changed file
  2. Extract: imports, exports, called functions, type references
  3. Diff against previous AST: what edges changed?
  4. petgraph traversal → impacted files (transitive)
  5. Parallel setxattr on ALL impacted files (dozens of calls)
  6. Append to edges.jsonl
  7. (optional) Update DuckDB analytical index

AGENT ASKS "what's impacted?" → read xattr → answer in microseconds

1.2 Language tiers

LayerLanguageWhy
Core FUSE daemon Rust Kernel integration (fuser 0.16.0), goroutine-free concurrency (tokio), petgraph 0.8.2 for graph traversal, tree-sitter in pure Rust, xattr crate, zero-GC trigger hot path, compile-time crash prevention
MCP server Python (pipx) Universal install, official duckdb package, os.getxattr built-in, no kernel integration needed
Go client library Go Thin wrapper — os.Getxattr via golang.org/x/sys/unix, DuckDB via marcboeker/go-duckdb, zero IPC for Go agents
CLI shim Rust (same binary) vfs read|write|meta|graph|mount — compiled into daemon, no subprocess
Import parsers Rust (tree-sitter) Go, Python, TypeScript, Rust, JavaScript, Java, C, C++, Ruby grammars — all pure Rust, zero FFI

2. Permission Model — FUSE-Enforced, Not Advisory

Core principle: The model sees real filesystem permission errors, not injected warnings. The VFS sets FUSE mode bits that the Linux kernel enforces. When the agent tries to write to a read-only file, it gets EACCES from the kernel — exactly like a real filesystem. No injected warnings, no special tool calls.

2.1 Kernel enforcement path

1. AGENT: echo "change" > /mnt/vfs/workspace/.gitignore
2. KERNEL: checks FUSE mode bits → file is 0444
3. KERNEL: returns -EACCES (Permission denied)
4. AGENT: "Permission denied" — the file isn't writable
5. MODEL: Learns that .gitignore is protected. Moves on.

2.2 Manifest permission rules

# .vfs/manifest.yaml
permissions:
  rules:
    # VFS internals — never writable
    - paths: [".vfs/**"]
      mode: 0444
    
    # Git internals — never writable through VFS
    - paths: [".git/**", ".gitignore", ".gitattributes", ".gitmodules"]
      mode: 0444
    
    # Dependency lock files — read-only (use package manager)
    - paths: ["**/go.sum", "**/Cargo.lock", "**/package-lock.json",
              "**/yarn.lock", "**/pnpm-lock.yaml"]
      mode: 0444
    
    # Generated code — read-only (edit the source, not the output)
    - paths: ["**/*.pb.go", "**/*.gen.go", "**/*.generated.*",
              "**/generated/**", "**/__pycache__/**"]
      mode: 0444
    
    # Vendored dependencies — read-only
    - paths: ["**/vendor/**", "**/node_modules/**"]
      mode: 0444
    
    # CI/CD config — read-only (change via PR, not direct edit)
    - paths: [".github/workflows/**", ".gitlab-ci.yml", "Dockerfile"]
      mode: 0444
    
    # Documentation — read-write
    - paths: ["docs/**", "*.md", "README.md"]
      mode: 0644
    
    # Source code — read-write (default workspace)
    - paths: ["src/**", "lib/**", "pkg/**", "cmd/**", "internal/**"]
      mode: 0644
    
    # Configuration files — read-write but no delete
    - paths: ["*.toml", "*.yaml", "*.yml", "*.json"]
      mode: 0644
      allow_delete: false

  default_mode: 0644

  # Backend-level overrides
  backends:
    - name: shared-lib        # mounted dependency repo
      mode: 0444              # entire mount read-only

3. Bubblewrap Sandboxing — Optional Security Layer

Verdict: Worth it as an OPTIONAL layer, not core. Bubblewrap wraps the agent's shell in a mount namespace. The VFS daemon runs on the host. The agent's process sees only the VFS mount. This protects against malicious escape, not just accidental damage.

3.1 Two layers of protection

LayerVFS permissionsBubblewrap
Enforced byLinux kernel (FUSE mode bits)Linux kernel (mount namespace)
ScopeWithin the VFS mount — agent can't write to .gitignoreOutside the VFS mount — agent can't see /etc, /home
Protects againstAccidental damage, policy violationsMalicious escape, host recon
Agent sees"Permission denied" on the fileFiles outside workspace simply don't exist
Runtime costZero (just mode bits)~100ms namespace creation per shell

3.2 Architecture

Host:                   warpfs-daemon (FUSE, full host access)
                            │
                    /mnt/vfs/workspace  ← mounted by daemon
                            │
Bubblewrap sandbox:    /workspace  ← bind-mounted read-write
                       / (root)    ← bind-mounted READ-ONLY
                       /tmp        ← isolated tmpfs
                       /proc       ← isolated
                            │
                       agent's shell (cd /workspace)

3.3 Manifest configuration

# .vfs/manifest.yaml
sandbox:
  enabled: false              # default: off
  engine: bubblewrap           # or: none
  isolate_network: true        # --unshare-net
  isolate_pid: true            # --unshare-pid
  read_only_root: true         # bind-mount / as read-only
  writable_paths:
    - /tmp
    - /workspace
Not core — optional security escalation. Bubblewrap adds 50MB package, ~100ms startup latency, and system-specific bind-mount configuration. The VFS permission model handles 95% of use cases. Bubblewrap is for untrusted agents or multi-tenant environments where agent escape is a real threat.

4. Multi-Repo Workspace

Core feature. Mount multiple git repos, S3 buckets, and local paths into one unified tree. Cross-repo graph edges. Agent opens one directory and sees everything. 
/mnt/vfs/workspace/          ← top-level VFS mount (not a git repo)
├── auth-service/            ← git@github.com:org/auth-service.git (main)
├── payment-service/         ← git@github.com:org/payment-service.git (develop)
├── shared-lib/              ← git@github.com:org/shared-lib.git (v2.1.0, read-only)
├── models/                  ← S3 bucket (not a repo)
├── docs/                    ← git@github.com:org/docs.git (main)
├── .vfs/                    ← TOP-LEVEL inventory (cross-repo graph)
│   ├── manifest.yaml        ← declares all repos & backends
│   ├── graph/edges.jsonl    ← CROSS-REPO edges
│   └── backends/mounts.yaml

  golib: { enabled: true }

# ── Multi-repo workspace ──

  - { name: shared-lib, url: git@github.com:org/shared-lib.git, ref: v2.1.0, writable: false }

# ── Backends ──
backends:
  s3:
    - { bucket: my-bucket, prefix: prod/, at: /project/models/, cache: /tmp/vfs-cache/, ttl: 3600, writable: false }

    - { path: /data/datasets/, at: /project/data/, writable: true }

# ── Metadata (xattrs set on every file) ──

    on_write: true
  impact: { enabled: true, max_depth: 5, parallel: true }
  cross_repo: true

# ── Permissions ──
permissions:

    - { paths: [".vfs/**", ".git/**", ".gitignore"], mode: 0444 }
    - { paths: ["**/go.sum", "**/Cargo.lock", "**/package-lock.json"], mode: 0444 }
    - { paths: ["**/*.pb.go", "**/*.gen.go", "**/generated/**"], mode: 0444 }
    - { paths: ["**/vendor/**", "**/node_modules/**"], mode: 0444 }

    - { name: shared-lib, mode: 0444 }

# ── Triggers (inotify-wired) ──
triggers:

7. The Agent Experience

# Agent opens workspace
$ cd /mnt/vfs/workspace

# Sees unified tree (6 repos + S3 + local path)
$ ls
auth-service/  payment-service/  shared-lib/  models/  docs/  datasets/

# Reads a file normally
$ cat auth-service/src/handler.go
(func Handler(...) — pristine file bytes from git backend)

# Asks: "what's connected to this file?"
$ getfattr -n user.vfs.relations auth-service/src/handler.go
user.vfs.relations="imports:auth-service/src/types.go,auth-service/src/middleware.go|external:shared-lib/pkg/utils.go"

# Asks: "what breaks if I change this?"
$ getfattr -n user.vfs.impact shared-lib/pkg/utils.go
user.vfs.impact="direct:shared-lib/pkg/auth.go|external:auth-service/src/handler.go,payment-service/src/checkout.rs"

# Tries to edit .gitignore
$ echo "*.log" > .gitignore
bash: .gitignore: Permission denied    ← kernel-enforced, not injected

# Tries to edit a generated file
$ echo "" > auth-service/src/auth.pb.go
bash: auth.pb.go: Permission denied    ← "edit the .proto file instead"

# Tries to edit a lock file
$ echo "" > go.sum
bash: go.sum: Permission denied        ← "use go mod tidy"

# Checks risk before editing
$ getfattr -n user.vfs.risk auth-service/src/middleware.go
user.vfs.risk="critical-path"          ← model: "be careful here"

# Checks freshness
$ getfattr -n user.vfs.last_test auth-service/src/login.go
user.vfs.last_tested="2026-06-14T09:33:14"
# Model compares with last_modified → STALE. Needs re-test.

8. Open Questions

  1. xattr namespace strategy? user.* is portable but per-UID. trusted.* requires CAP_SYS_ADMIN. Multi-UID agents need MCP proxy or trusted namespace.
  2. Metadata size limits? ext4 caps at 4KB per file. Highly-connected files may exceed this. Split across multiple xattrs or store heavy data in sidecar files.
  3. xattr sync across git clones? git clone doesn't preserve xattrs. Store in git notes or a sidecar file that vfs mount restores.
  4. Cross-filesystem xattrs? Moving files from ext4 to tmpfs = metadata loss. Detect and warn.
  5. Graph write amplification? Every file write triggers import re-parsing + edge updates + xattr writes + JSONL append. Measure overhead under load.
  6. Bubblewrap in Docker/CI? User namespaces are often disabled in containerized environments. Degrade gracefully.
  7. MCP vs direct xattr — which wins? MCP gives richer structured responses. getfattr is faster (no IPC). Both should work.
