🔑 The Key Insight

Preloop has already built the governance layer Helix needs — and it's open source. Helix shouldn't rebuild the MCP firewall, model gateway, or approval workflows. It should run Preloop as a sidecar and focus on what only Helix can do: the git forge, agent marketplace, prompt infrastructure, and agent-to-agent collaboration primitives.

✅ What Preloop Replaces for Helix

Instead of buying 4+ separate products (MintMCP for tool governance, Portkey/Helicone for model gateway, LangFuse for observability, custom Slack bots for approvals), Preloop ships all of it as one Apache 2.0 binary. Helix gets battle-tested enforcement without rebuilding it.

1. MCP Firewall — The Core Interception Layer

🎯 The Trick

Preloop doesn't modify any agent. Instead, it subclasses FastMCP to create DynamicFastMCP — an MCP server that the agent connects to instead of the real MCP server. The agent thinks it's talking to a normal MCP server. Preloop is actually in the middle, evaluating every tool call against access rules before forwarding.

Execution Flow

Agent → POST /mcp/v1 → PreloopBearerAuthBackend
↓
DynamicFastMCP._call_tool() → policy_evaluator → _evaluate_rule_candidates()
First matching rule wins (priority-ordered). No match = ALLOW (audited).
Action: DENY
Return error to agent
with rule description
Action: REQUIRE_APPROVAL
Notify Slack/Mobile/Webhook
Agent waits or polls async
Action: ALLOW
Execute tool
Return result to agent

Access Rules — YAML + CEL

# Policy-as-code, stored in account metadata, evaluated at runtime
version: "1.0"
metadata:
  name: "Production Safeguards"

tools:
  - name: "bash"
    source: mcp
    approval_workflow: "deploy-approval"
    justification: required
    conditions:
      - expression: "args.command.contains('deploy') && args.command.contains('production')"
        action: require_approval
      - expression: "args.file_path.endsWith('.env')"
        action: deny

Subject-Scoped Governance

Every tool call carries a subject context (API key ID, managed agent ID). The scope chain resolves: API key scope → managed agent scope → account defaults. Each scope can have its own allowed_models, model_budgets, tool_rules, and tool_enabled_overrides — stored as JSONB in account metadata.

💡 What to Borrow

The DynamicFastMCP subclassing pattern is clean and reusable. CEL for policy conditions is standard and well-supported. The priority-ordered, first-match-wins rule model is correct. The subject-scope chain resolution (subject_scope_chain() in subject_governance.py) is elegant and Helix should adopt it.

⚠️ What to Build Differently for Helix

Preloop stores policies in account meta_data JSONB — no version history, no audit trail for policy changes. Helix needs proper agent_policies tables with versioning. Preloop's tool filtering is account-level; Helix needs per-repo, per-branch scoping. Helix should add AGENTOWNERS files (like CODEOWNERS) to auto-assign approval to specific humans.

2. Model Gateway — Proxy + Budget + Secret Custody

🎯 The Trick

Preloop runs an OpenAI-compatible and Anthropic-compatible API that agents connect to instead of the real provider API. Behind the scenes, it proxies to the real provider using LiteLLM — but only after budget checks, attribution tagging, and credential resolution. Agents never see real API keys.

Request Flow

Agent → POST /openai/v1/chat/completions → Bearer: short-lived runtime token
↓
authenticate_bearer_token() → ModelGatewayAuthContext
↓
budget_enforcer.check_budget() → BLOCK or → ALLOW
↓
resolve_ai_model_runtime() → Model → Provider → Credentials
↓
get_secret_service().resolve() → API key from vault/local_encrypted
↓
LiteLLMModelGatewayBackend.completion() → Real provider API
↓
estimate_cost() → ApiUsage ledger → GatewayUsageSearchService → realtime dashboard update

Key Design Decisions

LiteLLM as Backend
100+ providers, one API

Preloop maps provider_name to LiteLLM's routing. DeepSeek → deepseek, Qwen → OpenAI-compatible, Bedrock → AWS. Handles streaming, OAuth, and OAuth token injection for Claude Code and Codex CLI.

Secret Custody
Agents never see real API keys

Provider credentials stored in SecretReference (local_encrypted or Vault/OpenBao). Agents receive short-lived preloop_rt_* bearer tokens. Revocation is instant.

Budget Enforcement
Pre-flight, not post-hoc

ModelGatewayBudgetService checks account/flow/subject limits before upstream dispatch. Blocks the call if over budget. No post-hoc surprises.

ApiUsage Ledger
Canonical cost tracking

Every model call writes one row: tokens in/out, estimated cost, provider, model, account, runtime principal, session ID. Single source of truth for all cost analytics.

💡 What to Borrow

The gateway-as-proxy pattern with short-lived tokens is perfect for Helix. LiteLLM is battle-tested. The ApiUsage canonical ledger is the right design. Budget check pre-dispatch prevents runaway spend. OAuth passthrough for Claude Code/Codex CLI is essential.

⚠️ What to Build Differently for Helix

Helix needs per-repo/per-PR budget scoping — "this PR has $5 budget, stop when exceeded." Helix needs model fallback chains: try v4-pro → on budget exhaust → v4-flash → on 404 → step-3.5-flash. Preloop's cost analytics are shallow in OSS; Helix should build session-value analysis into core. Pricing should pull live from OpenRouter API daily.

3. Agent Enrollment — Config Rewrite Pattern

🎯 The Trick

preloop agents discover finds agent configs on disk, imports MCP servers and model metadata into Preloop's DB, mints a managed runtime identity, and then rewrites the agent's local config files to point to Preloop instead of the original endpoints. The agent doesn't know it's been enrolled — it just starts routing through Preloop.

Hermes Enrollment Flow

# One-command discovery and onboarding
preloop agents discover          # Scans ~/.hermes/config.yaml

preloop agents onboard hermes    # Creates ManagedAgent + credential
preloop agents install-plugin hermes  # Installs runtime plugin
preloop agents validate hermes   # Verifies plugin connects

The config rewrite is conservative — it adds Preloop as an MCP server, rewrites the model endpoint to the gateway, and preserves all existing MCP servers. Two-layer enrollment: CLI config gives MCP/gateway routing immediately; runtime plugin adds Agent Control later.

💡 What to Borrow

The config rewrite pattern is battle-tested for Hermes specifically. The ManagedAgent + ManagedAgentCredential + RuntimeSession data model is clean. Two-layer enrollment (CLI + plugin) is the right separation.

⚠️ What to Build Differently for Helix

Helix should manage agents per repo, not just per account. An agent enrolled for reports should have different tools than one for helix-core. Config should be declarative and version-controlled in .helix/agent-policies.yaml in the repo. Helix needs batch enrollment for Kubernetes namespaces.

4. Agent Control — Live Bidirectional Channel

🎯 The Trick

A persistent WebSocket from the agent back to Preloop lets a human operator send commands (including voice) directly into the agent's runtime session. The agent's plugin receives the command and injects it as the next user turn — audited, governed, and routed through the same MCP firewall and model gateway.

Connection Manager

class AgentControlConnectionManager:
    """In-process registry for connected managed agents."""
    _connections: dict[str, WebSocket]       # connection_id → WebSocket
    _agent_connections: dict[str, str]       # agent_id → connection_id
    _connection_agents: dict[str, str]       # connection_id → agent_id

    # New connection for same agent evicts old one.
    # Heartbeats every 15s. Missed heartbeat → disconnect.

Command Lifecycle

Operator sends command → REST API → Persist to DB (audited)
↓
Publish to NATS → Deliver via WebSocket (if connected)
↓
Plugin receives send_message envelope → Inject as next user turn in agent
💡 What to Borrow

The plugin-based architecture for Agent Control — don't modify Hermes core, ship a runtime plugin. The preloop.control config block pattern is declarative and versionable. WebSocket + NATS for horizontal scaling. Heartbeat + reconnect + command persistence for durability.

⚠️ What to Build Differently for Helix

Helix needs Agent Control scoped to git operations: "review PR #42", "fix CI failure in commit abc123", "rebase onto main." Helix should integrate with git webhooks — new PR automatically notifies the assigned agent. Voice is secondary; Helix's primary Operator surface should be the PR interface itself.

5. Borrow, Extend, Build — The Full Matrix

BORROW — Integrate directly or steal the architecture

FeatureWhyHow for Helix
Preloop as a whole Already built the governance layer. Apache 2.0. Works with Hermes. Run as sidecar. Helix focuses on forge + marketplace + prompt infra.
MCP Firewall pattern DynamicFastMCP + CEL + subject-scoped governance. Proven code. Steal architecture. Extend with repo-scoped rules.
LiteLLM as gateway 100+ providers, streaming, OAuth. Preloop's wrapper is thin. Use directly. Add Helix-specific budget/fallback layers.
Agent enrollment rewrite Hermes adapter handles ~/.hermes/config.yaml. One command. Run Preloop's CLI. Extend with batch enrollment.
Subject-scope chain API key → managed agent → account. Simple priority chain. Adopt pattern. Add repo scope: repo → team → account.

BUILD — From scratch, only Helix can do this

Git Forge
Forgejo-based

Preloop is governance middleware, not a forge. Helix IS the forge — the place where code lives, PRs happen, and agents work alongside humans.

Agent Marketplace
Discoverability & reputation

No agent registry in Preloop. Helix needs discoverable agent catalog with per-agent reputation, custom definitions, and one-click install.

Prompt Infrastructure
Versioning, CI/CD, replay

Preloop governs model calls, not prompt engineering. Helix needs prompt-as-code, versioned prompts, PromptFoo evaluation in CI.

Agent-to-Agent PR Review
Bidirectional collaboration

Preloop governs tool access. It doesn't facilitate agent-to-agent negotiation. Helix's agent-to-agent PR review is a new primitive.

Git-Level Agent Control
PR-scoped commands

Preloop's Agent Control is generic. Helix needs git-aware: "review PR #42", "fix CI in abc123", "rebase onto main."

Agent Attestation
Signed agent commits

Not in Preloop's model. Helix needs signed agent commits with model+prompt provenance per line of code.

🔌 The Adapter — ~2,000 lines of Python

A thin translation layer that maps Helix's repo-scoped policies into Preloop's subject-scoped governance. Reads .helix/policies.yaml from each repo, translates AGENTOWNERS into approval routing, maps PR scope to budget enforcement, and verifies agent attestation on commit. This is the only new code Helix needs for governance — everything else is Preloop.
