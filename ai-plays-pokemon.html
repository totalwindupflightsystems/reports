▸ What This Actually Is

An AI plays Pokémon Red/Blue on a real emulator — no pre-programmed logic, no game-specific hacks. The AI sees screenshots through a vision model, reasons about what to do next via a thinking model, and presses buttons through tool calls. Every decision is a real LLM inference.

What's different from other "AI plays Pokémon" projects

  • Not RL-trained — zero pre-training on the game
  • Not hardcoded — no if-then rules for battles or navigation
  • Not pixel-reading heuristics — the vision model actually sees the screen
  • Tool-calling architecture — structured commands, not text parsing
  • Stacked prompt configs per game generation — adapts to GB vs GBA UI differences
  • Two-model split: vision is cheap+precise, thinking is free

Why two models

Vision models that can read screenshots are expensive ($0.15–$15/M tokens). Text-only models that make decisions are cheap or free ($0.00/M). Splitting them means the expensive model runs once per screenshot (~$0.005), and the free model makes all the actual gameplay decisions.

Decision Loop (per tick, ~1/sec)

  1. Capture — pygba renders frame, auto-crops SGB border for GB games
  2. Vision — Gemma 12B reads screenshot → structured JSON (screen_type, enemy, HP%, text, menu)
  3. Select — screen_type + game generation → choose prompt stack (gen1/battle.yaml etc.)
  4. Assemble — Stack 5 layers: system prompt + tool definitions + vision data + memory + examples
  5. Decide — owl-alpha receives assembled prompt + tool schema → returns tool call
  6. Execute — tool call → emulator button press with frame-accurate timing
  7. Remember — record action, update memory (last 20 actions, party status, goal)

▸ Stacked Prompt System

Prompts are layered configs selected by game generation × screen type. The vision model's output determines which stack loads. Each stack has 5 layers.

Config Structure

configs/prompts/
├── gen1/              # Red/Blue/Yellow
│   ├── battle.yaml    # FIGHT→move flow, B/W HP bars
│   ├── overworld.yaml # D-pad, tall grass, NPC trainers
│   ├── menu.yaml      # Start menu, item use, save
│   └── dialog.yaml    # NPC text, yes/no prompts
└── gen3/              # FireRed/LeafGreen
    ├── battle.yaml    # Colored HP, abilities, held items
    ├── overworld.yaml # Running shoes, berry trees
    ├── menu.yaml      # Bag pockets, TM/HM system
    └── dialog.yaml    # Teachy TV, rival interactions

5-Layer Stack (each YAML)

system
Role, rules, game context, current objective
tools
Available commands: press_button, wait, combo
observation
Vision output injected: enemy, HP, text, menu
memory
Recent actions, party status, active goal
examples
2–3 few-shot tool call examples for this screen type

▸ Vision Prompt Design (The Hardest Part)

The vision model must extract specific actionable data — never describe the scene. The prompt forbids narration entirely. Output is JSON only.

What It Extracts

  • screen_type: battle | overworld | menu | dialog | title
  • enemy_pokemon: Pokémon name from battle screen
  • player_hp_pct / enemy_hp_pct: 0–100 from HP bar fill
  • text_lines: verbatim text from any visible text boxes
  • menu_items: visible options (FIGHT, BAG, POKéMON, RUN)
  • adjacent_info: what tiles surround the player
  • status_icons: PAR, SLP, PSN, BRN, FRZ
  • dialog_prompt: yes/no question text
