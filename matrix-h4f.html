Matrix for H4F

Self-hosted Matrix chat as a third channel for Hermes agent access — alongside Telegram and email. One shared homeserver, per-friend private rooms, session isolation via the existing Hermes gateway.

Design Decision

One homeserver, not seven

One conduwuit instance (~100MB RAM) serves all friends. User isolation via Matrix rooms, not separate processes.

Design Decision

One bot account, N rooms

@hermes joins every friend's private room. Hermes session DB keys on room_id — zero cross-contamination.

Design Decision

Friends create rooms

No auto-room-creation in provision_friend.py. Friends get credentials, log into Element, create room, invite @hermes. Bot auto-joins.

Provisioning Flow (per friend)

Matrix registration becomes step 4 in the existing pipeline. One API call to conduwuit — no new infrastructure.

1

Docker containers EXISTING

hermes + dind containers. Unchanged.

2

OpenRouter key + guardrail EXISTING

API key provisioned, assigned to Flash/Plus/Pro guardrail. Unchanged.

3

Nextcloud email account EXISTING

Email address provisioned. Will now also deliver Matrix credentials.

4

Matrix user registration NEW

POST to conduwuit's register endpoint. Auto-generated password stored in friend's .env.

POST /_matrix/client/v3/register
→ @friendname:hermes-matrix.local
→ auto-generated password
→ access_token stored in known-friends.json
5

known-friends.json MODIFIED

Two new fields per friend entry — no schema break, additive only.

"matrix_user_id": "@sift:hermes-matrix.local"
"matrix_password_hash": "..."
6

Welcome delivery MODIFIED

Matrix credentials included in welcome email. Friend self-serves room creation.

Session Isolation

The Matrix adapter already handles this — identical to how Telegram works today. One bot, many isolated sessions.

Friend Channel Hermes Session Isolation Mechanism
Wojons Private room (!room_wojons) Session A room_id key
Sift Private room (!room_sift) Session B room_id key
Trixie DM with @hermes Session C room_id key
Room-level

auto_thread: true

Every bot response spawns a thread. Threads are isolated from the main room timeline. Threads the bot participates in don't require @mention.

Room-level

dm_auto_thread: true

DMs auto-create threads without needing @mention. 1:1 conversations get automatic thread isolation.

Room-level

require_mention: true

In shared rooms, bot only responds to @hermes mentions. Prevents cross-talk noise. In DMs, no mention needed.

Session DB

Keyed on (platform, room_id, thread_id)

Zero cross-friend contamination. Friend A's session state is cryptographically separate from Friend B's.

What Doesn't Change

Gateway

One process, all platforms. Matrix is just another adapter — no new gateway instance.

Container model

One hermes+dind stack per friend. Unchanged. Matrix connects through the gateway, not the container.

Token model

Friends still bring their own Telegram bot token. Matrix is additive, not a replacement for Telegram.

Cron pipeline

5-layer pipeline unchanged. Matrix health check added to doctor but pipeline stays the same.

Explicitly Out of Scope

v1 E2EE for the bot — conduwuit supports cross-signing but bot key management adds significant complexity
v1 Federation with other homeservers — this is internal-only
v1 Matrix as primary channel — Telegram remains primary, Matrix is the self-hosted option
v1 Room auto-creation — friends self-serve room creation and bot invitation
v1 Per-friend homeservers — one shared conduwuit serves everyone
