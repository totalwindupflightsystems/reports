Verdict: Conduwuit for closed mode. Synapse if federation matters.

Conduwuit wins on every axis except federation-at-scale. If we enable federation — so friends with existing matrix.org accounts can message our bot without creating a new account — Synapse's battle-tested federation stack becomes the tiebreaker. Conduwuit federates but hasn't been through the same fire as Synapse with thousands of interconnected servers.

3. The Federation Question

This is the architectural fork. It determines which homeserver we choose, what infrastructure we need, and what the friend onboarding experience looks like.

★ Recommended for v1

Closed — No Federation

Every friend gets a new account on our homeserver during provisioning. No existing Matrix accounts — clean slate.

  • Zero infrastructure beyond a Docker container
  • No DNS, no TLS, no public exposure
  • conduwuit: SQLite, 100 MB RAM, one process to manage
  • Backup: one directory. Migration: copy the data dir
  • Friends with existing Matrix accounts need a second one
  • "Another account to manage" friction for power users
$0/mo additional (co-hosted on H4F VPS)
v2 / Federation-needed

Open — Federation Enabled

Friends can message @hermes from their existing matrix.org (or any) account. Or create a local one. Maximum flexibility.

  • Zero friction for friends who already use Matrix
  • "Just DM @hermes:matrix.h4f.example" — no account creation
  • Hybrid: local accounts still available for non-Matrix friends
  • Synapse + PostgreSQL — heavier stack, more maintenance
  • Public domain + TLS cert required
  • Federation moderation: spam, abuse vectors
  • conduwuit federation less tested at scale
~$12–20/mo (separate VPS or upgraded H4F host)

4. The "Attach" Model — Friends With Existing Matrix

If a friend already has @theirname:matrix.org and doesn't want a second account, here's how they connect — depending on which mode we run.

Friend's situation Closed mode Federated mode
No Matrix account Provisioning creates @friend:hermes-matrix.local. They log into Element pointed at our server. Same experience as new email account. Provisioning creates @friend:matrix.h4f.example OR they register on matrix.org and DM our bot. Either works.
Has matrix.org account Must create a second account on our homeserver. Two Matrix accounts, two clients (or one client with two logged-in sessions). Friction. They DM @hermes:matrix.h4f.example from their existing matrix.org client. No new account. Federation routes the message.
Runs their own homeserver Same as above — second account required on our server. They add our server to their federation allowlist. Their users DM our bot directly. Enterprise-grade.

Progressive migration path

Start closed (conduwuit, no federation). If enough friends complain about the second account, migrate to Synapse with federation. conduwuit's data directory is a RocksDB — migration tools exist for conduwuit → Synapse, and the reverse is possible with the conduwuit import tool. The closed mode doesn't lock us in — it's the v1 that keeps options open for v2.

6. Decision Matrix

If... Then...
None of your friends use Matrix today Conduwuit, closed, co-hosted. Zero cost, zero complexity. Friends get accounts at provisioning time — same flow as email.
1–2 friends already have Matrix and would resist a second account Conduwuit, closed, co-hosted. The friction of a second account for 1–2 people is lower than the infrastructure cost of federation. They can run two sessions in Element.
Multiple friends actively use Matrix and refuse second accounts Synapse, federated, separate $12 VPS. Federation is the only way to meet them where they are. Bite the PostgreSQL bullet.
You want federation but refuse PostgreSQL Conduwuit, federated, with monitoring. It works — conduwuit federates. It just hasn't been through matrix.org-scale fire. For a private friend group (not public rooms), the federation load is negligible.
You want to start simple, keep options open Conduwuit, closed, co-hosted. Migrate later if needed. Conduwuit's RocksDB is portable. Migration tools exist. v1 is not a trap.

One thing you can't defer: public DNS

If there's any chance you'll enable federation later, register the domain NOW and point it at the server. Matrix user IDs include the server name (@user:matrix.h4f.example), and changing the server name later requires a database wipe — every user loses their account. You can run closed on a public domain. You can't add a domain later without nuking accounts.

7. Recommendation

V1 — Ship Now

Conduwuit, closed, co-hosted

Add compose/matrix/ to the H4F repo. One docker compose up -d. provision_friend.py registers Matrix users alongside email. Zero new cost. Zero new services.

Use a real domain as server name from day one (not localhost) — preserves the federation upgrade path without account wipes.

V2 — If Federation Needed

Synapse, federated, separate VPS

Spin up a $12/mo CX32. Migrate conduwuit data. Enable federation. Friends on matrix.org DM @hermes:matrix.h4f.example without creating accounts.

Trigger: >2 friends already on Matrix AND unwilling to create second accounts.
