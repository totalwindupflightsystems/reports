Hermes VFS

Virtual Filesystem for Agentic Workflows — Architecture & Design Reference
Status: Research / Design Phase  |  Date: June 2026

1. Core Concept

Clone a repo → Mount it with Hermes VFS → Point your agent workspace at the mount.

Agents read and write files like normal. They never see the VFS. Behind the scenes, the VFS injects context, resolves virtual paths, tracks relationships between files, and federates storage across git, S3, and local disk — all invisible to the agent.

# Agent runs:
$ cat /mnt/vfs/project/src/auth/login.go

# VFS intercepts, injects context, returns:
[=== Context: Auth Module ===]
Related: src/auth/types.go, src/auth/middleware.go
Test coverage: 87% (login_test.go)
Last modified: 2 hours ago

[=== File: login.go ===]
package auth
func Login(...) { ... }

# Agent processes normally — never knows context was injected.

3.1 FUSE Deep Dive

FUSE (Filesystem in Userspace) forwards kernel VFS syscalls to your userspace program via /dev/fuse. Every open(), read(), write(), stat() becomes a message your program handles.

How it works:

  1. Kernel receives open("/mnt/vfs/src/main.go")
  2. Kernel checks: this path is a FUSE mount → forward to userspace daemon via /dev/fuse
  3. Your daemon receives a FUSE_OPEN request with the inode number
  4. Your daemon resolves inode → git blob, reads content, returns file handle
  5. Kernel returns fd to the calling process — agent sees a normal file descriptor

Go library: hanwen/go-fuse is the current best. Actively maintained, v2 API, supports both low-level "raw" mode and higher-level "nodefs" mode.

// Minimal FUSE read-only filesystem
import "github.com/hanwen/go-fuse/v2/fs"
import "github.com/hanwen/go-fuse/v2/fuse"

type VFSFile struct {
    fs.Inode
    content []byte
}

func (f *VFSFile) Read(ctx context.Context, dest []byte, 
                        off int64) (fuse.ReadResult, syscall.Errno) {
    end := int(off) + len(dest)
    if end > len(f.content) { end = len(f.content) }
    return fuse.ReadResultData(f.content[off:end]), 0
}

func (f *VFSFile) Getattr(ctx context.Context, fh fs.FileHandle, 
                           out *fuse.AttrOut) syscall.Errno {
    out.Size = uint64(len(f.content))
    out.Mode = 0644
    return 0
}

3.2 9P Deep Dive — The FUSE Alternative

9P is the Plan 9 file protocol. Simpler than FUSE because it uses a single kernel module (v9fs) for ALL 9P mounts — you write a userspace server, and the kernel's existing 9P client handles the mount.

9P Advantages
  • No custom kernel module per filesystem
  • Protocol is ~15 message types (FUSE has ~40+)
  • WSL2 uses 9P internally for Windows/Linux file sharing
  • Go has aqwari.net/net/styx (9P server library)
  • Each file operation = simple request/response
9P Limitations
  • Less tooling than FUSE
  • mmap, sendfile don't work over 9P
  • Permissions model is simpler than POSIX ACLs
  • Not all Linux distros enable v9fs by default

3.3 CLI Shim — The Pragmatic Start

Instead of a kernel filesystem, expose the VFS as commands. Agents call vfs read instead of cat. This is the fastest path to a working prototype.

$ vfs read src/auth/login.go
[=== Context: Auth Module ===]
Related: src/auth/types.go, src/auth/middleware.go
Test coverage: 87%

[=== File: login.go ===]
package auth
func Login(...) { ... }

$ vfs ls src/auth/
login.go
types.go
middleware.go
README.md
login_test.go

$ vfs graph deps src/auth/login.go
src/auth/login.go → src/auth/types.go (imports)
src/auth/login.go → src/auth/middleware.go (imports)
src/auth/login.go ← src/auth/login_test.go (tested_by)

$ vfs mount s3://my-bucket/models/ /project/models/
Mounted s3://my-bucket/models/ at /project/models/
For the prototype: Start with CLI shim. It proves the concept (context injection, graph queries, virtual mounts) with zero kernel involvement. Graduate to 9P or FUSE once the design is proven. CLI shim also works in Docker containers and CI pipelines where FUSE mounts are restricted.

4. Layer 2: Metadata Engine — The Graph Problem

Files don't exist in isolation. auth/login.go imports auth/types.go. handler.go is tested by handler_test.go. Three files form a logical feature. The metadata engine tracks these relationships and makes them queryable.

4.1 Option A: SQLite Adjacency (Recommended)

Two tables. Simple, queryable with SQL, exportable to git.

CREATE TABLE files (
    id INTEGER PRIMARY KEY,
    path TEXT UNIQUE NOT NULL,
    hash TEXT NOT NULL,          -- SHA-256
    size INTEGER, mtime INTEGER,
    metadata JSON                -- {"feature": "auth", "owner": "alex"}
);

CREATE TABLE edges (
    from_id INTEGER REFERENCES files(id),
    to_id INTEGER REFERENCES files(id),
    relation TEXT NOT NULL,       -- 'imports', 'tested_by', 'documents'
    weight REAL DEFAULT 1.0,
    UNIQUE(from_id, to_id, relation)
);

Key queries:

-- What does auth/login.go import?
SELECT f.path FROM files f 
JOIN edges e ON e.to_id = f.id 
WHERE e.from_id = (SELECT id FROM files WHERE path = 'src/auth/login.go')
  AND e.relation = 'imports';

-- What breaks if I delete types.go? (reverse dependency)
SELECT f.path FROM files f 
JOIN edges e ON e.from_id = f.id 
WHERE e.to_id = (SELECT id FROM files WHERE path = 'src/auth/types.go');

-- Show me the full auth module (recursive dependency tree)
WITH RECURSIVE deps AS (
  SELECT to_id, 1 AS depth FROM edges 
  WHERE from_id = (SELECT id FROM files WHERE path = 'src/auth/login.go')
  UNION ALL
  SELECT e.to_id, d.depth + 1 FROM edges e 
  JOIN deps d ON e.from_id = d.to_id WHERE d.depth < 10
)
SELECT DISTINCT f.path, d.depth FROM files f 
JOIN deps d ON f.id = d.to_id ORDER BY d.depth;

✓ Strengths

• Single file, zero setup
• SQL — everyone knows it
• ~10M edges before perf degrades
• Export to .vfs/edges.jsonl → commit to git
• Recursive CTEs for graph queries

✗ Limitations

• Recursive CTEs slow at depth > 5 with millions of edges
• No native property graph model
• Schema changes need migrations
• Not ideal for > 50M edges

4.2 Option B: Property Graph (Dgraph / Badger-native)

Files as nodes, relationships as edges with typed predicates. Native graph traversal.

// Conceptual schema (GraphQL+-like)
type File {
    path: string @index(exact)
    hash: string
    feature: string
}
// Reverse edge: files that test this file
type TestedBy {
    from: File    // the test file
    to: File      // the source file
}

// Query: all files in auth feature, with their test files
{ files(func: eq(feature, "auth")) { 
    path
    tests: ~tested_by { path }
}}

When to use: You need multi-hop queries regularly AND have > 1M files. For most repos (< 100k files), SQLite is faster and simpler.

4.3 Option C: Content-Addressable Overlay

Store relationships by content hash, not file path. When a file changes, its hash changes → new edge file. Old relationships persist (describe the file at that point in time).

.vfs/graph/edges/
  sha256:abc123def → {"imports": ["sha256:789...", "sha256:012..."],
                       "tested_by": ["sha256:345..."],
                       "feature": "auth-module"}
  sha256:789ghi012 → {"imported_by": ["sha256:abc..."],
                       "feature": "auth-module"}

Why it's elegant: File renames don't break relationships (hash stays same). Multiple versions coexist. Natural fit with git's content-addressable model. Flat files — no database.

Where it hurts: Every file change = new hash = new edge file. Query requires: hash → read edge file → resolve hashes to paths. Not queryable with SQL.

4.4 Option D: Git-Native (git notes)

Git already stores a DAG. git notes attaches metadata to any git object without modifying it.

git notes add -m "edge:imports→sha256:def456" sha256:abc123
git notes add -m "feature:auth-module" sha256:abc123
git notes show sha256:abc123
# → edge:imports→sha256:def456
#   feature:auth-module

When: You absolutely need metadata versioned alongside code AND the repo is small (< 10k files). Useful as a transport between clones, even if the query engine is SQLite.

Virtual Mounts

The manifest declares what storage backs each path. The VFS routes transparently.

# .vfs/manifest.yaml
mounts:
  s3://my-bucket/models/:
    at: /project/models/
    cache: /tmp/vfs-cache/models/
    ttl: 3600
    
  git+ssh://git@github.com/org/other-repo.git:
    at: /project/vendor/other-lib/
    ref: main

The agent sees /project/models/bert-large.bin as a normal file. The VFS routes to S3, caches locally, and serves from cache on subsequent reads.

6. The Manifest — Triggers, Injection, Rules

The manifest (.vfs/manifest.yaml) defines everything the VFS does beyond basic file serving.

# .vfs/manifest.yaml
version: 1

global:
  inject:
    - when: "*.go"
      prepend_template: |
        // === Context ===
        // Related: {{ .RelatedPaths }}
        // === End Context ===
  import_parsing:
    enabled: true
    languages: [go, python, typescript, rust]

paths:
  "src/auth/":
    inject:
      - when: "*.go"
        sidebar: "src/auth/README.md"
      - when: "handler_*.go"
        context_query:
          relations: [imports, tested_by]
          max_depth: 2
          max_results: 10
    graph:
      auto_edges: true
      feature_tag: "auth-module"
    
  ".env":
    inject: []              # Never inject into secrets
    graph: { exclude: true }

triggers:
  - name: "lint-on-save"
    when: "*.go"
    on: write
    run: "golangci-lint run {{ .FilePath }}"
    timeout: 30s
    
  - name: "update-graph"
    when: "*"
    on: [write, delete]
    run: "vfs graph update --path {{ .FilePath }}"

virtual:
  "/project/.vfs/deps/auth-module.md":
    query:
      root: "src/auth/login.go"
      relations: [imports, tested_by]
      direction: downstream
      format: markdown

  "/project/.vfs/impact/types.go.md":
    query:
      root: "src/auth/types.go"
      relations: [imported_by]
      direction: upstream
      format: markdown

mounts:
  s3://my-bucket/models/:
    at: "/project/models/"
    cache: "/tmp/vfs-cache/"
    ttl: 3600

Injection Flow — Step by Step

1. AGENT: cat /mnt/vfs/project/src/auth/login.go

2. VFS INTERCEPTS:
   ├── Resolve path → git backend: blob sha256:abc123
   ├── Read file content
   ├── Check manifest for "src/auth/login.go":
   │   ├── global.inject["*.go"] → prepend context template
   │   ├── paths["src/auth/"].sidebar → read README.md
   │   ├── paths["src/auth/"].context_query → SQL recursive CTE
   │   │   → [types.go, middleware.go, login_test.go, session.go]
   │   └── paths["src/auth/"].auto_edges → parse imports
   │       → upsert edges in SQLite
   └── Return: prepend + sidebar + file_content

3. AGENT RECEIVES:
   [=== Context: Auth Module ===]
   Related: types.go, middleware.go, login_test.go, session.go
   Test coverage: 87%
   
   [=== Auth Module README ===]
   # Auth Module
   Handles OAuth2 + session tokens...
   
   [=== File: login.go ===]
   package auth
   func Login(...) { ... }

8. Full Workflow — Clone → Mount → Work

# 1. Clone the repo
$ git clone https://github.com/org/project.git ~/projects/myapp

# 2. Initialize VFS
$ cd ~/projects/myapp
$ vfs init
  → Created .vfs/manifest.yaml
  → Created .vfs/graph/ directory

# 3. (Optional) Edit manifest
$ vim .vfs/manifest.yaml

# 4. Mount through VFS
$ vfs mount ~/projects/myapp /mnt/vfs/myapp
  → Backend: git
  → Metadata: SQLite (.vfs/graph.db)
  → Interface: FUSE
  → Mounted at /mnt/vfs/myapp

# 5. Point agent at the mount
$ export WORKSPACE=/mnt/vfs/myapp
$ agent run "Add OAuth to auth module"

# Agent sees:
#   cat /mnt/vfs/myapp/src/auth/login.go
#   → File + injected context
#
#   ls /mnt/vfs/myapp/.vfs/deps/
#   → auth-module.md (computed dependency graph)
#
#   cat /mnt/vfs/myapp/project/models/bert-large.bin
#   → Routed to S3, cached locally

# 6. Unmount
$ vfs unmount /mnt/vfs/myapp
  → Exported graph to .vfs/edges.jsonl
  → Committed .vfs/ to git
  → Unmounted

10. Open Questions

  1. FUSE vs 9P vs CLI for prototype? CLI fastest. 9P gives real cat/ls. Decision affects agent compatibility.
  2. Import parsing: regex or tree-sitter? Regex fast but fragile. Tree-sitter accurate but heavier.
  3. Graph sync across machines? Git notes could transport edges between clones.
  4. Injection overhead at scale? Every read() = manifest check + graph lookup. Caching needed above 100 req/s.
  5. Write path? Auto-update edges on write? Or background job?
  6. Binary files? Skip injection for binary MIME types.
  7. Security? Should VFS strip secrets from .env before serving? Refuse certain paths?
  8. Latency? FUSE round-trip ~50μs + metadata engine query time. Fine for agents, not for HFT.
