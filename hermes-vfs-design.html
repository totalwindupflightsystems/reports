Hermes VFS

Virtual Filesystem for Agentic Workflows — Architecture & Design Reference
Status: Research / Design Phase  |  Date: June 2026  |  Version: 2 — Metadata-First Architecture

⚠️ Design Correction (v1 → v2): We initially considered injecting context directly into file content on read — prepending headers, appending related-file lists. This is wrong. The model would treat injected text as part of the actual file and act on it as if it were source code. Instead, we use filesystem-native extended attributes (xattrs) to store metadata alongside files without modifying file content. When agent tools don't expose xattrs, we provide an MCP server that gives the agent explicit tool calls to query metadata. The file content stays pristine.

1. Core Concept

Clone a repo → Mount it with Hermes VFS → Point your agent workspace at the mount.

Files stay exactly as they are. Metadata — relationships, graph edges, backend routing, version info — lives in filesystem extended attributes (xattrs) and inventory files. The agent queries metadata explicitly through native tools or an MCP server. File content is never modified.

2. Architecture — Three Layers

┌─────────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────────┘

3. The Key Insight: Metadata, Not Injection

When an agent reads a file, the file content must be exactly what's on disk. If we prepend "Related: types.go, middleware.go" into the bytes of login.go, the agent will think those are actual lines of code. It'll try to parse them, edit them, commit them.

Instead: metadata lives alongside the file in the filesystem's native metadata layer — extended attributes (xattrs). The file bytes are untouched. The agent queries metadata separately, explicitly, through tools it already has or through an MCP server we provide.

3.1 Extended Attributes (xattrs) — The Linux Native Metadata Layer

Every major Linux filesystem supports extended attributes — key-value pairs attached to files and directories, invisible to normal reads, queryable with standard tools.

Filesystemxattr SupportLimitsNotes
ext4Full — user.*, trusted.*, system.* namespacesMax 4KB per file (one block), no per-attr limitEnabled by default on most distros
XFSFullMax 64KB per fileBetter for metadata-heavy workloads
btrfsFullMax 4KB per fileAlso supports subvolumes/snapshots
ZFSFullMax 64KB per fileAlso supports datasets, snapshots, send/receive
tmpfsNo—In-memory only, no xattr support
NFS v4Yes (named attributes)Server-dependentRequires NFS v4.1+ for full support
9P (Plan 9)Yes (via stat extensions)Protocol limitsWSL2's 9P has partial xattr support

3.2 How Agents Read Metadata

Path A: Native tools (if the agent has them)

# Read a single xattr
$ getfattr -n user.vfs.relations /mnt/vfs/project/src/auth/login.go
# → user.vfs.relations="imports:src/auth/types.go,src/auth/middleware.go|tested_by:src/auth/login_test.go"

# Read all VFS xattrs on a file
$ getfattr -d -m "user.vfs.*" /mnt/vfs/project/src/auth/login.go
# → user.vfs.relations="..."
#   user.vfs.feature="auth-module"
#   user.vfs.backend="git"
#   user.vfs.hash="sha256:abc123def..."

# List all files with a specific xattr value
$ find /mnt/vfs/project -xattr -name "*.go" -exec sh -c 'getfattr -n user.vfs.feature "$1" 2>/dev/null | grep -q auth-module && echo "$1"' _ {} \;

Path B: MCP server (always available, pipx/npx install)

When the agent's tools don't expose xattrs (most AI coding tools don't), we provide an MCP server. The agent gets explicit tool calls:

# MCP tools exposed to the agent
Tool: vfs_get_metadata(path: string) → Metadata
  Returns all xattrs for a file: relations, feature tags, backend info, hash, etc.

Tool: vfs_query_graph(query: GraphQuery) → GraphResult
  Runs graph queries: "what imports X", "show auth module", "impact of changing Y"

Tool: vfs_list_virtual(path: string) → DirectoryListing
  Lists virtual directories — including paths that don't exist in git
  (S3 buckets, remote repos, computed views)

Tool: vfs_resolve_backend(path: string) → BackendInfo
  Returns which backend stores this file, cache status, sync state

# Install (pipx for Python, npx for Node)
$ pipx install hermes-vfs-mcp
$ hermes-vfs-mcp --mount /mnt/vfs/myproject

# Or via npm
$ npx hermes-vfs-mcp --mount /mnt/vfs/myproject
Why MCP is the right fallback: It's protocol-standard, already supported by Hermes Agent and other AI tools. The agent doesn't need filesystem-specific knowledge — it just calls tools. The MCP server translates tool calls into xattr reads, graph queries, and backend lookups. Works identically whether the underlying filesystem is ext4, XFS, or NFS.

4. Inventory Files — The Source of Truth

xattrs are great for per-file metadata, but they're not queryable at scale — you can't "find all files in the auth feature" without walking the entire tree. Inventory files solve this. They're structured files stored in .vfs/ that the VFS loads at mount time and keeps in memory (or SQLite) for fast queries.

4.1 Inventory File Types

FilePurposeFormatExample
.vfs/graph/edges.jsonl File relationship graph — imports, tests, docs, dependencies JSONL — one edge per line, append-only {"from":"src/auth/login.go","to":"src/auth/types.go","rel":"imports"}
.vfs/backends/mounts.yaml Virtual mount table — maps paths to storage backends YAML S3 bucket → /project/models/, remote git → /project/vendor/lib/
.vfs/features/tags.yaml Feature/module groupings — which files belong to which logical unit YAML auth-module: [src/auth/*.go, docs/auth/*.md]
.vfs/blobs/index.jsonl Content-addressed blob index — path → hash → backend location JSONL {"path":"models/bert.bin","hash":"sha256:...","backend":"s3","cache":"hit"}
.vfs/manifest.yaml Master config — triggers, rules, metadata namespaces, default settings YAML Global rules, path-specific configs, trigger definitions
.vfs/triggers/hooks.yaml File event hooks — what runs on write, delete, rename YAML on:write "*.go" → run: "golangci-lint"

4.2 The Edges Inventory — Graph in a Flat File

# .vfs/graph/edges.jsonl
{"from":"src/auth/login.go","to":"src/auth/types.go","rel":"imports","weight":1.0}
{"from":"src/auth/login.go","to":"src/auth/middleware.go","rel":"imports","weight":1.0}
{"from":"src/auth/login_test.go","to":"src/auth/login.go","rel":"tested_by","weight":1.0}
{"from":"src/auth/middleware.go","to":"src/auth/login.go","rel":"imported_by","weight":1.0}
{"from":"src/auth/login.go","to":"docs/auth/README.md","rel":"documented_by","weight":0.5}

Why JSONL: Append-only. Git-friendly (diffs cleanly). Can be streamed — don't need to load the whole file. Can be split across multiple files (edges-001.jsonl, edges-002.jsonl). Easy to generate from import parsers, test runners, linters.

At mount time: VFS loads edges into SQLite for queryable graph operations. The JSONL file is the source of truth (versioned in git). SQLite is the query cache (rebuildable).

# On vfs mount:
1. Read .vfs/graph/edges.jsonl (and any edges-N.jsonl files)
2. INSERT or REPLACE into SQLite graph DB
3. Build indexes: from_id, to_id, relation
4. Ready for queries

# On edge change:
1. Append new line to edges.jsonl
2. Upsert into SQLite
3. (Optionally) git commit the new edge

4.3 Virtual Folders — Paths That Don't Exist in Git

The backends inventory maps paths to storage providers. A folder can appear in the file tree that has no corresponding directory in git — the VFS synthesizes it from the backend.

# .vfs/backends/mounts.yaml
mounts:
  # S3 bucket appears as a local directory
  s3://my-project-models/prod/:
    at: /project/models/
    region: us-east-1
    cache: /tmp/vfs-cache/models/
    ttl: 3600
    writable: false
    
  # Another git repo mounted inside this one
  git+ssh://git@github.com/org/shared-lib.git:
    at: /project/vendor/shared-lib/
    ref: main
    cache: /tmp/vfs-cache/shared-lib/
    
  # Local directory from outside the repo
  /data/datasets/:
    at: /project/data/
    writable: true

# What the agent sees:
$ ls /mnt/vfs/project/models/
bert-large-v2.bin    # Actually in S3
gpt2-medium.bin       # Actually in S3

$ ls /mnt/vfs/project/vendor/shared-lib/
go.mod                # Actually in a different git repo
utils.go              # Actually in a different git repo

# Metadata tells the truth:
$ getfattr -n user.vfs.backend /mnt/vfs/project/models/bert-large-v2.bin
# → user.vfs.backend="s3://my-project-models/prod/bert-large-v2.bin"

$ getfattr -n user.vfs.cache_status /mnt/vfs/project/models/bert-large-v2.bin
# → user.vfs.cache_status="hit"

Auto-upload tracking: When an agent writes to a virtual folder backed by S3, the VFS intercepts the write, uploads to S3, updates the blob index, and records the new hash in xattrs. The agent thinks it wrote a local file. The VFS handled the upload transparently.

# Agent writes a file
$ cp new-model.bin /mnt/vfs/project/models/new-model.bin

# VFS intercepts:
1. Write to local cache: /tmp/vfs-cache/models/new-model.bin
2. Upload to S3: s3://my-project-models/prod/new-model.bin
3. Record in blob index: {"path":"models/new-model.bin","hash":"sha256:...","backend":"s3"}
4. Set xattrs: user.vfs.backend="s3://...", user.vfs.hash="sha256:..."
5. Append to .vfs/blobs/index.jsonl
6. Return success to agent

# Agent has no idea this happened.

5. The MCP Server — Metadata as Tool Calls

The MCP server is the bridge between "filesystem has metadata" and "agent can query metadata." It reads xattrs, inventory files, and the SQLite graph DB, and exposes them as structured tool calls.

5.1 Complete MCP Tool Set

# ── File Metadata ──
vfs_get_metadata(path: string, keys?: string[])
  → { path, xattrs: { key: value, ... }, size, mtime, hash, backend }

vfs_set_metadata(path: string, key: string, value: string)
  → { success, previous_value? }

# ── Graph Queries ──
vfs_graph_related(path: string, relations?: string[], max_depth?: int)
  → { files: [{ path, relation, depth }], total }

vfs_graph_impact(path: string, max_depth?: int)
  → { dependents: [{ path, relation, depth }] }
  # "What breaks if I change this file?"

vfs_graph_module(module: string)
  → { files: [path, ...], edges_count, test_coverage_pct }

vfs_graph_untested()
  → { files: [path, ...] }
  # "Which source files have no tests?"

vfs_graph_stats()
  → { total_files, total_edges, most_connected: [{path, degree}], orphans: [path] }

# ── Virtual Directory Operations ──
vfs_list_directory(path: string)
  → { entries: [{ name, type, backend?, size?, virtual: bool }] }

vfs_resolve_path(path: string)
  → { real_path, backend, cached, sync_status }

# ── Backend Operations ──
vfs_backend_status(path: string)
  → { backend, cache_hit, cache_path, remote_url, last_synced }

vfs_sync_backend(path: string)
  → { synced_files: int, errors: [string] }

5.2 MCP Server Implementation

# Python MCP server (pipx installable)
# hermes-vfs-mcp/server.py

from mcp.server import Server, Tool
import os, json, sqlite3

server = Server("hermes-vfs")

MOUNT_POINT = os.environ.get("HERMES_VFS_MOUNT", "/mnt/vfs")

def get_xattr(filepath, key):
    """Read a single extended attribute."""
    try:
        return os.getxattr(filepath, f"user.vfs.{key}").decode()
    except OSError:
        return None

def get_all_vfs_xattrs(filepath):
    """Read all user.vfs.* xattrs on a file."""
    try:
        raw = os.listxattr(filepath)
        return {
            k.replace("user.vfs.", ""): os.getxattr(filepath, k).decode()
            for k in raw if k.startswith("user.vfs.")
        }
    except OSError:
        return {}

@server.tool("vfs_get_metadata")
def vfs_get_metadata(path: str, keys: list[str] = None):
    full_path = os.path.join(MOUNT_POINT, path.lstrip("/"))
    xattrs = get_all_vfs_xattrs(full_path)
    if keys:
        xattrs = {k: v for k, v in xattrs.items() if k in keys}
    stat = os.stat(full_path)
    return {
        "path": path,
        "xattrs": xattrs,
        "size": stat.st_size,
        "mtime": stat.st_mtime,
    }

@server.tool("vfs_graph_related")
def vfs_graph_related(path: str, relations: list[str] = None, max_depth: int = 2):
    db = sqlite3.connect(f"{MOUNT_POINT}/.vfs/graph.db")
    # Recursive CTE for graph traversal
    rel_filter = ""
    if relations:
        placeholders = ",".join(["?" for _ in relations])
        rel_filter = f"AND e.relation IN ({placeholders})"
    
    query = f"""
    WITH RECURSIVE related AS (
        SELECT e.to_id, e.relation, 1 AS depth
        FROM edges e JOIN files f ON e.from_id = f.id
        WHERE f.path = ?
        UNION ALL
        SELECT e.to_id, e.relation, r.depth + 1
        FROM edges e JOIN related r ON e.from_id = r.to_id
        WHERE r.depth < ?
    )
    SELECT DISTINCT f.path, r.relation, r.depth
    FROM related r JOIN files f ON r.to_id = f.id
    ORDER BY r.depth, f.path
    """
    params = [path, max_depth]
    if relations:
        params = [path] + relations + [max_depth]
    
    rows = db.execute(query, params).fetchall()
    return {
        "files": [{"path": r[0], "relation": r[1], "depth": r[2]} for r in rows],
        "total": len(rows)
    }

# ... remaining tools ...

if __name__ == "__main__":
    server.run()

6. Manifest — Master Configuration

The manifest (.vfs/manifest.yaml) ties everything together. It defines metadata namespaces, triggers, backend mounts, and graph behavior.

# .vfs/manifest.yaml
version: 2  # metadata-first architecture

# ── Metadata namespaces ──
# Each namespace becomes a user.vfs. xattr
metadata:
  namespaces:
    - relations     # imports, tested_by, documented_by
    - feature       # logical module grouping
    - backend       # storage backend (git, s3, local)
    - hash          # content hash (sha256)
    - origin        # where this file came from (git ref, s3 key, etc.)

# ── Graph configuration ──
graph:
  inventory: .vfs/graph/edges.jsonl
  auto_discover:
    enabled: true
    languages: [go, python, typescript, rust]
    on_read: true       # parse imports when file is first read
    on_write: true      # re-parse imports when file is written
  
  sqlite_cache: .vfs/graph.db
  max_edges_per_file: 100000  # split into edges-N.jsonl after this

# ── Feature tags ──
features:
  auth-module:
    paths: [src/auth/]
    tags: [security, critical-path]
  payment-module:
    paths: [src/payments/, src/billing/]
    tags: [pci, high-risk]
  docs:
    paths: [docs/]
    tags: [documentation]

# ── Backend mounts ──
mounts:
  s3://my-project-models/prod/:
    at: /project/models/
    region: us-east-1
    cache: /tmp/vfs-cache/models/
    ttl: 3600
    
  git+ssh://git@github.com/org/shared-lib.git:
    at: /project/vendor/shared-lib/
    ref: main

# ── Triggers ──
triggers:
  - name: lint-go
    when: "*.go"
    on: write
    run: "golangci-lint run {{ .FilePath }}"
    timeout: 30s
    
  - name: update-graph
    when: "*"
    on: [write, delete]
    run: "vfs graph update --path {{ .FilePath }}"
    
  - name: sync-s3
    when: "/project/models/*"
    on: write
    run: "vfs backend sync --path {{ .FilePath }}"

# ── Path-specific metadata defaults ──
paths:
  "src/":
    metadata:
      feature: "auto"     # derive from parent directory
      backend: "git"
      
  "src/generated/":
    graph: { auto_discover: false }   # don't parse generated code
    
  ".env":
    metadata:
      visibility: "restricted"   # never expose xattrs on secrets
    graph: { exclude: true }
    triggers: []

7. Complete Data Flow

7.1 Agent Reads a File (with Metadata)

1. AGENT: cat /mnt/vfs/project/src/auth/login.go

2. VFS (FUSE/9P):
   ├── Read from git backend: blob sha256:abc123
   ├── Return EXACT file bytes (no injection)
   └── Done. Agent has pristine file content.

3. AGENT (separately, or via MCP): getfattr -n user.vfs.relations login.go
   OR: mcp__vfs_get_metadata(path="src/auth/login.go")

4. VFS / MCP:
   ├── Read xattr: user.vfs.relations → "imports:types.go,middleware.go|..."
   ├── (Optional) Query SQLite graph for deeper relationships
   └── Return structured metadata

5. AGENT now knows:
   - File content (pristine, from step 2)
   - File relationships (from step 4)
   - What module it belongs to, what tests it, where it's stored
   Agent can use this however it wants — the metadata is explicit, not hidden.

7.2 Agent Writes to a Virtual Folder (S3-backed)

1. AGENT: write /mnt/vfs/project/models/new-model.bin

2. VFS intercepts:
   ├── Check: this path matches mount "s3://my-project-models/prod/"
   ├── Write to local cache: /tmp/vfs-cache/models/new-model.bin
   ├── Compute hash: sha256:def789...
   ├── Upload to S3: PUT s3://my-project-models/prod/new-model.bin
   ├── Set xattrs:
   │   user.vfs.backend = "s3://my-project-models/prod/new-model.bin"
   │   user.vfs.hash = "sha256:def789..."
   │   user.vfs.cache_status = "synced"
   ├── Append to .vfs/blobs/index.jsonl
   ├── Trigger (if configured): notify CI, run validation
   └── Return success to agent

3. AGENT sees: file written successfully. Nothing unusual.

8. Interface Layer — FUSE / 9P / CLI / MCP

InterfaceFile ReadsMetadata AccessSetupBest For
FUSE Real open()/read() — any tool getfattr, listxattr syscalls Kernel module Full POSIX transparency
9P Real mount — cat, ls work 9P stat extensions Userspace server Simpler than FUSE, WSL2-native
CLI shim vfs read, vfs write vfs meta, vfs graph Zero Fastest prototype, containers
MCP server N/A — agent uses normal file tools vfs_get_metadata, vfs_graph_* pipx/npx install Always works, any platform
Go library vfs.Read() vfs.GetXattr() Zero Go agents, no IPC overhead
Recommended approach: Start with CLI shim for the prototype (fastest to working code). Layer MCP server on top immediately (so any agent can use it). Graduate to FUSE or 9P when the design is proven and you need real cat/ls behavior. MCP is the permanent fallback — it works everywhere including Docker, CI, and Windows.

10. Open Questions

  1. xattr namespace strategy? user.vfs.* is portable. But should we use trusted.* for system-level metadata that agents shouldn't modify? security.* for access control?
  2. Metadata size limits? xattrs have per-filesystem limits (4KB on ext4, 64KB on XFS). Graph edges for highly-connected files could exceed this. Split across multiple xattrs? Store heavy data in sidecar files?
  3. xattr sync across clones? git clone doesn't preserve xattrs. Should we store xattr data in git notes or a sidecar file that vfs mount restores?
  4. MCP vs direct xattr — which wins? If an agent supports both getfattr and MCP, which path is preferred? MCP gives richer structured responses. getfattr is faster (no IPC).
  5. Graph write amplification? Every file write potentially triggers import re-parsing + edge updates + xattr writes + JSONL append. Acceptable overhead?
  6. Binary files and xattrs? Should we attach metadata to binary files (models, images)? The xattrs work the same, but the metadata would be manually curated, not auto-discovered.
  7. Cross-filesystem xattrs? What happens when a file moves from an xattr-supporting filesystem (ext4) to one that doesn't (tmpfs)? Metadata loss. Detect and warn?
  8. Multi-user xattr permissions? user.* xattrs are per-user. If two agents run as different users, they won't see each other's xattrs. Use trusted.* (requires CAP_SYS_ADMIN)?
