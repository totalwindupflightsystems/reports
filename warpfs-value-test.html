What Worked

✅ Accurate — 61/61 importers verified

WarpFS said: 61 files import internal/db.

Grep confirmed: exactly 61.

Spot-checks on harness.go, main.go, and server.go all matched. The graph engine correctly identifies semantic imports, not string occurrences in comments or log messages.

✅ Architecture surfaced in seconds

From three CLI commands, the agent learned:

  • Hub-and-spoke architecture: internal/db is the central package with 61 importers
  • Peer packages: API, MCP, harness, tools, session, hitl — all import internal/db but not each other
  • Leaf packages: billing, security, secrets have zero internal dependents
  • Test relationships: WarpFS detected server_test.go → tests edge for both api and mcp

This would normally require opening ~20 files and tracing imports manually.

✅ Metadata filtering beats grep ambiguity

Query: "Show me the API gateway file."

WarpFS: Tagged internal/api/server.go with user.vfs.role=api-gateway. Query returns exactly that file.

Grep: grep -rl 'func.*Server|NewRouter|HandleFunc' returns test helpers, mock servers, any struct named Server. Ambiguous.

✅ Transitive impact — BFS traversal

Query: "What's the blast radius of changing internal/db?"

WarpFS: graph impact --max-depth 3 — instant BFS traversal across 61 direct dependents. Each dependent's dependents also traced. Full transitive tree in one command.

Without: grep for importers → open each file → grep for their importers → repeat. Manual BFS across 61+ files. Minutes of work.

What Didn't Work

❌ Go parser doesn't scale

166 files (Consensus): ~3 seconds — fine.

866 files (Hivemind): timed out at 120s — never completed.

The Go AST parser needs optimization for 500+ file repos. Without this, WarpFS can't handle larger Go projects.

timeout 120 warpfs graph discover  →  [TIMEOUT — 866 Go files]
❌ MCP tool consistency gaps

CLI works. MCP has issues:

  • vfs_list_directory returned empty results (CLI equivalent works)
  • vfs_graph_related through MCP returned incorrect format — 0 files import net/http when CLI shows 48
  • vfs_resolve_path only worked after code fix (fallback to cwd)

For WarpFS to be agent-usable, the MCP server must be as reliable as the CLI. Agents connect through MCP.

⚠️ No auto-classification

WarpFS can't automatically tag cmd/consensus/main.go as the entrypoint. A human (or prior agent) must run warpfs meta --set user.vfs.role entrypoint.

Value requires metadata investment upfront. The graph is automatic; semantic tagging is manual.

⚠️ Only 1 language detected in multi-lang project

Parser supports 9 languages, but graph discover reported "1 languages" for Consensus. Rust files (.rs), configs, and other non-Go files weren't parsed. Only Go imports were captured.

The parser works (verified in multi-language test files), but graph discover only walks the current language by default.

Verdict: Worth It, With Caveats

The core idea — "query code structure without reading code" — is real and demonstrated.

The dependency graph is the killer feature. 1014 edges in 3 seconds, perfectly accurate, enabling instant architecture insight and transitive impact analysis. Metadata tagging eliminates grep ambiguity for role/status queries.

The gaps are fixable: Go parser optimization, MCP tool consistency pass, and auto-classification heuristics. None are fundamental flaws — they're Phase 4 polish items.
