What Coding Hermes IS

  • End-to-end SDLC: Each foreman tick runs a complete software development lifecycle — self-heal → discover → analyze → plan → implement → verify → commit → learn → scan → repeat.
  • Memory-driven: DuckBrain stores decisions, pitfalls, and patterns per project. Off-by-One provides cross-project solution caching. No tick starts from scratch.
  • Provider-aware: Foremen stay on PAYG (always reachable, always rescuable). Workers burn prepaid flat-rate buckets. When one bucket exhausts, the foreman auto-switches.
  • Quality-gated: GitReins guard (secrets, build, lint, tests) runs before every commit. GitReins judge (LLM evaluation) verifies acceptance criteria. No "it works" claims without evidence.
  • Self-healing: Supervisor detects broken crons, zombie states, orphans, duplicates — fixes without asking. Foremen fix transient CI failures, environment issues, identity problems before touching tasks.

What Coding Hermes IS NOT

  • Not a CI system: Runs on schedule, not on push. Inspects, plans, codes — not just builds and tests.
  • Not a chatbot: Foremen don't chat. Workers don't wait. Autonomous from tick to commit.
  • Not a replacement for Bane: Worker → Foreman → Supervisor → Bane. Each level exhausts its own capabilities. Bane provides strategy and architecture.

The 12-Step Loop in Detail

0 Self-Heal

Fix before touching tasks. Verify git identity (totalwindupflightsystems). Pull dependencies. Check CI health. Fix transient failures immediately. Validate language-specific environment. If fundamentally broken, create INFRA task and skip to 1.6 — don't burn ticks fighting the environment.

1 Read Board

Read .coding-hermes/tasks.md. Count pending ## [ ] headers and - [ ] subtasks. FIFO task selection — pick the oldest pending task. Never cherry-pick. If HIGH_PRIORITY label exists, pick that first. Board empty? → 1.5 Discovery Sweep.

1.5 Discovery Sweep Board Empty

Ordered priority stack:

  1. Build integrity: make build / go build ./... — does it compile?
  2. Usability testing: Hit live endpoints — do they respond correctly?
  3. Spec alignment: grep TODO/FIXME/HACK/XXX — what's spec'd but not built?
  4. Doc audit: Run README commands — do they work?
  5. CI audit: gh run list — any non-transient failures?

Max 5 tasks per sweep. If genuinely empty after sweep (valid outcome), commit findings, write DuckBrain, scan signals, end tick.

2 Hilo Impact Analysis

hilo graph → spatial map. hilo impact → what depends on this? hilo classify → refactor, feature, bug, or infra? Pass transitive impact to the worker: "Modify parser.go — depends_on: lexer.go, ast.go, formatter.go. Risk: high."

3 DuckBrain Context Load

Query 4 domains: decisions (what was chosen and why), pitfalls (what went wrong last time), patterns (reusable solutions), status (current project state). Summarize for the worker — don't dump raw output.

4 Pre-Load

Assemble 8-element context package: task description, Hilo impact, DuckBrain context, relevant file contents, acceptance criteria, verification requirements, commit instructions, GitReins instructions. Compile through prompt-foundry. Worker gets ONE prompt — no skills, no architecture, no fleet context.

5 Spawn Worker NOT delegate_task

hermes chat -q '<prompt>' -m '<model>' --provider '<bucket>' --background. Independent session — worker gets its own model and provider on prepaid flat-rate. Foreman monitors every ~60s. Stuck after 15 min → kill and retry with different model/provider.

6 GitReins Guard Tier 1

Secrets, build, lint, tests.

ResultAction

FAIL — transientRetry once. Still failing? Worker should have caught it.

7 GitReins Judge Tier 2

LLM evaluation: does the code meet acceptance criteria?

ResultAction
PASSProceed to Step 8 (commit)
FAIL — minor gapsWorker fixes specific gaps. One more pass.
FAIL — fundamentalTask too big or underspecified. Break into subtasks. Next tick picks up.

8 Commit

git add <specific files> — never git add -A. git diff --cached before committing. Message format: type: description — why. Addresses TASK-ID. Types: feat, fix, refactor, test, docs, ci, infra, chore. --no-verify (guard already ran). Verify post-commit: git log -1, git status --short.

1.6 Scan External Signals Bridge between ticks

Before the next tick, scan what changed:

  1. CI: gh run list — new failures? Previously-failing now green?
  2. Remote commits: git fetch && git log HEAD..origin/main — external pushes?
  3. New issues: gh issue list — bug/critical labeled → create task immediately.
  4. Dependency updates: go list -u / pip list --outdated / npm outdated — security updates?
  5. Cross-project DuckBrain: did another foreman discover a pattern relevant here?

Max 3 tasks from signals. "No external changes" is valid — log in DuckBrain.

Provider Architecture

💡 The golden rule: Foremen MUST be on PAYG (deepseek-foreman). Always reachable, always rescuable — add credit, unblock instantly. Workers burn prepaid flat-rate buckets. When one bucket locks, the foreman (still alive on PAYG) auto-switches workers to the next available plan.
RoleModelProvider TypeWhy
Foremandeepseek-v4-proPAYGMust always load. PAYG always unblockable via credit.
Foreman (idle)deepseek-v4-flashPAYGSame rule — presence non-negotiable.
Supervisordeepseek-v4-flashFlat-rate (opencode-go)Light work. Bucket locks → fleet healing pauses, foremen keep running.
Worker (coding)glm-5.2 / MiniMax-M3 / gpt-5.6 / grok-4.5 / kimi-k2.7Prepaid flat-rateBurn quota coding. Locks when exhausted — foreman auto-switches.
⚠️ Foremen on prepaid = project dark. Locked prepaid bucket → foreman can't load → can't rebalance → can't spawn workers → dead until billing cycle resets. PAYG is always rescuable.
⚠️ Workers spawn via hermes chat CLI, NOT delegate_task. delegate_task inherits the foreman's PAYG provider — workers burn the tracked key. hermes chat -q -m <model> --provider <bucket> gives the worker its own prepaid bucket.

Key Pitfalls

🪤 Foremen on prepaid = project dark. Locked bucket → can't load → can't spawn → dead. PAYG is rescuable.
🪤 Workers via delegate_task burns PAYG key. Use hermes chat -q -m <model> --provider <bucket> for independent sessions.
🪤 Never set model/provider on no_agent jobs. $100+ burned on DuckBrain sync crons. Identify by skills array, not name.
💡 Worker bucket exhaustion is normal. Foreman's job is provider switching. zai-glm locked → switch to minimax → locked → switch to xai-oauth.
💡 Pre-existing guard failures don't block progress. Flag them, move forward. Don't let old problems hold new work hostage.
💡 FIFO task selection. Always. Pick the oldest pending task. 3+ ticks stuck on same task? Escalate to supervisor — task may be too big.
✅ Off-by-One is cross-project memory. A solution for ASCE might solve the same problem for Helios. Submit completed tasks, discover before next tick.
