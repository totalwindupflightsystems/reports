What Coding Hermes IS

  • Autonomous: Foremen wake on schedule, scan task boards, spawn workers, verify results, commit code — zero human intervention per tick.
  • Self-healing: Supervisor detects broken cron schemas, zombie states, orphaned projects, and duplicate foremen — fixes them without asking.
  • Self-rebalancing: Supervisor adjusts foreman speed and worker models based on pending task count and commit activity.
  • Memory-driven: Every project has a DuckBrain namespace storing decisions, pitfalls, and patterns across ticks. Foremen recall before coding and write after committing.
  • Quality-gated: GitReins guards run on every commit. Ad-hoc verification scripts are mandatory after every code edit. No verbal "it works" claims accepted.

What Coding Hermes IS NOT

  • Not a CI system: It doesn't run on push. It runs on schedule — inspecting, planning, and coding autonomously.
  • Not a chatbot: Foremen don't chat. They scan, spawn, verify, commit. Workers write code. Neither waits for human input during a tick.
  • Not a replacement for Bane: The escalation path is Worker → Foreman → Supervisor → Bane. Each level exhausts its own capabilities first. Bane provides strategy and architecture — the fleet executes.
  • Not container-dependent: Foremen work directly on the host. No Docker sandboxes, no container orchestration. The host IS the runtime.

The Escalation Hierarchy

flowchart TD
    WORKER["Worker (spawn) — 90% of work"]
    WORKER -->|"stuck"| FOREMAN["Foreman (per-project) — 10%"]
    FOREMAN -->|"can't resolve"| SUP["Supervisor (fleet) — 1%"]
    SUP -->|"can't resolve"| BANE2["Bane — strategy, architecture"]

Worker → writes code, runs tests, commits, fixes own compile/test errors.
Foreman → preloads context, breaks down complex tasks, picks worker model, resolves conflicts.
Supervisor → rebalances models/speeds, fixes schedule corruption, creates foremen, detects orphans.
Bane → strategy, new capabilities, architecture decisions.

Each level must NOT escalate what it can handle itself. A worker that hits a compile error fixes it — doesn't ask the foreman. A foreman that sees a rate limit switches providers — doesn't wait for the supervisor.

The 10-Step Foreman Loop

Every foreman tick runs this exact flow. The foreman inspects and delegates — it NEVER writes code itself.

0
Self-Heal — Verify git identity, pull dependencies, check CI health. Fix transient failures before touching any tasks.
1
Read Board — Read .coding-hermes/tasks.md. Count pending tasks. If empty, jump to 1.5 Discovery Sweep.
1.5
Discovery Sweep — Find work: spec alignment (what's spec'd but not built?), doc audit, CI audit, endpoint usability testing. Queue up to 5 new tasks.
2
Hilo Impact — hilo graph, hilo impact, hilo classify. Understand blast radius before touching code.
3
DuckBrain Recall — Load project memory: past decisions, pitfalls, patterns. Context the worker wouldn't otherwise have.
4
Pre-Load — Assemble context package: task + Hilo impact + DuckBrain memory + relevant files + acceptance criteria. Compile through prompt-foundry.
5
Spawn Worker — delegate_task(goal=..., context=..., role="leaf"). Worker model selected by task complexity. Worker gets ONE prompt — no skills.
6
GitReins Guard — Tier 1 static checks: secrets, build, lint, tests. Must pass before commit.
7
GitReins Judge — Tier 2 LLM evaluation. Does the code actually meet the acceptance criteria?
8
Commit — Targeted add only (never git add -A). Correct authorship. Descriptive message.
9
Off-by-One — Submit the problem to the pre-solve lab. Discover cached solutions for the next tick.
10
DuckBrain Write — Store findings: what worked, what didn't, what pitfalls to avoid next time.

Supervisor Phases

Runs every 4 hours. Heals the fleet, audits health, rebalances resources, and reports.

Phase 0 — Auto-Heal (runs first, before any audit)

Sub-PhaseWhat It Fixes
0A. SchemaMissing expr, kind: "every" → interval, mismatched schedule_display
0B. Statestate: completed → scheduled, accidentally paused → re-enabled
0C. OverdueJobs that missed their schedule → force-fire immediately
0D. OrphansProjects with task boards but no foreman → auto-create foreman
0E. DuplicatesTwo foremen on same workdir → pause legacy one
0F. Missing BoardsForeman exists but no tasks.md → create bootstrap board

Phase 2 — Fleet Audit

Sub-PhaseWhat It Checks
2A. FiringDid every foreman fire on time? ✅ 🟡 🔴
2B. ErrorsWhy didn't it fire? State machine bug, scheduler crash, disabled?
2C. ResultsWhat did it produce? Git commits, task progress, CI status?
2D. RebalanceSpeed up active projects, slow down idle ones. Never touch pinned projects.

Phase 5 — Fleet Report

Self-contained Python script → HTML report with fleet table, metrics, what-got-done section → pushed to GitHub Pages.

Phase 6 — CI Check

Batch-fetch CI status for all projects. Classify: passing / failing / queued / billing-blocked.

Model & Provider Architecture

The golden rule: Foremen must ALWAYS be reachable. If a foreman can't load because its provider is locked, the entire project goes dark — no rebalancing, no worker spawning, nothing. Workers do the heavy coding and burn through prepaid quota. The foreman (always alive on PAYG) watches and rebalances workers across available plans.

RoleModelProvider TypeWhy
Foremandeepseek-v4-proPAYG (deepseek-foreman)Must always load. If locked, entire project goes dark — no rebalancing, no spawning. PAYG is always unblockable: add credit, fix it instantly.
Foreman (idle budget)deepseek-v4-flashPAYG (deepseek-foreman)Same rule — foreman presence is non-negotiable.
Supervisordeepseek-v4-flashFlat-rate (opencode-go)Fleet management is light work. If bucket locks, fleet healing pauses but foremen keep running.
Worker (heavy)glm-5.2 / MiniMax-M3 / gpt-5.6Prepaid flat-rateBurn through quota getting coding done. If one bucket locks, foreman switches workers to another available plan.
Worker (light)grok-4.5 / kimi-k2.7Prepaid flat-rateBug fixes, small changes. Same bucket-switching logic.
Worker (budget)deepseek-v4-flashFlat-rate (opencode-go)Idle project, minor work.
💡 The strategy: Workers burn prepaid buckets — get as much coding done as possible on flat-rate plans. When a bucket locks (prepaid plans DO lock when exhausted), the foreman (still alive on PAYG) auto-switches workers to the next available prepaid plan. Foreman is on PAYG because you can always add credit to unblock it — can't do that with a locked prepaid bucket.
⚠️ Foremen on prepaid is a failure mode. If a foreman's prepaid bucket locks, the foreman can't load. Can't load = can't rebalance = can't spawn workers = project is dead until the billing cycle resets. PAYG is always rescuable: add credit, foreman comes back online, resumes work.

Autonomous coding loops. ALWAYS on PAYG — deepseek-foreman provider. Skills: coding-hermes-foreman, coding-hermes-cron, hilo-usage, gitreins. Supervisor can rebalance speed and worker models, but NEVER changes the foreman provider away from PAYG.

Load-bearing. If they fail, foremen lose context across ticks.

Key Pitfalls

🪤 Foremen on prepaid = project goes dark. If a foreman's prepaid bucket locks, it can't load. Can't load = can't rebalance = can't spawn workers = dead until billing cycle resets. Foremen MUST be on PAYG (deepseek-foreman). You can always add PAYG credit to unblock — can't un-lock a prepaid bucket.
🪤 Never change a foreman's model field to a coding model. The foreman inspects and delegates — it stays on v4-pro or v4-flash. Coding models (glm-5.2, MiniMax-M3, etc.) are for workers only.
🪤 Never set model/provider on no_agent jobs. $100+ burned when supervisor set deepseek-v4-pro on 54 DuckBrain sync/data loader/watchdog jobs. These run scripts, not LLMs. Identify by skills array, not name keywords.
💡 Worker bucket switching is the foreman's job. When a worker's prepaid bucket locks (zai-glm exhausted), the foreman auto-switches workers to the next available plan (minimax, xai-oauth, kimi). Foreman stays alive on PAYG the whole time.
💡 Missing schedule.expr crashes the ENTIRE scheduler. Not just that one job — ALL cron execution stops. This is a JSON schema bug, fix immediately.
