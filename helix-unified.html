💡 This Is the Key Pattern

The developer edits .helix/policies.yaml in their repo and pushes. Helix detects the change via git webhook, parses the file, and calls Preloop's REST API to create/update the corresponding ToolAccessRule, ApprovalWorkflow, and ManagedAgentAIModelBinding records. No one logs into Preloop. No one sees Preloop's policy UI. The repo is the source of truth.

🔑 The Contract

Helix never touches Preloop's database directly. Helix never renders Preloop's frontend. All configuration flows through Preloop's REST API. All enforcement flows through Preloop's MCP firewall and model gateway. Helix queries Preloop for cost data, audit logs, and agent status — then renders them in its own UI. Preloop is PostgreSQL. Preloop is NATS. Preloop is infrastructure.

Key Workflows — What Happens When

1. Developer Pushes a New .helix/policies.yaml

1
Git push triggers Helix webhook
Developer commits .helix/policies.yaml to repo and pushes. Forgejo fires a push webhook to Helix Core.
2
Helix parses and validates the policy file
The .helix/ config parser validates YAML syntax, checks that referenced agents exist in the registry, verifies branch names match the repo's actual branches, and resolves AGENTOWNERS to actual user IDs.
3
Helix Governance Adapter translates to Preloop API calls
Each policy rule becomes a Preloop REST API call. The adapter handles batching, error recovery, and idempotency (same config pushed twice = same Preloop state).
Helix → Preloop Translation
bash on feat/* → allow
→ POST /api/v1/access-rules
{action: "allow", condition: "args.branch.matches('feat/*')", priority: 10}
Preloop Response
201 Created
Rule stored in tool_access_rule table.
Active immediately for next tool call.
4
Policy is live — agents governed immediately
The next time helix-reviewer makes a tool call, Preloop's MCP firewall evaluates the new rules. No restart. No redeploy. Policy-as-code, pushed to git, enforced in real time.

2. Developer Enrolls a New Agent

1
Developer clicks "Enroll Agent" in Helix UI
Selects agent type (Hermes, Claude Code, Codex CLI), names it, assigns it to repos, sets initial budget.
2
Helix calls Preloop to create the managed agent
Helix creates:
• Agent Registry entry (agent marketplace ID)
• Repo permissions (which repos this agent can access)
• Budget allocation ($/day/repo)
Helix calls Preloop:
POST /api/v1/agents/onboard
PUT /api/v1/managed-agents/{id}/bindings
PUT /api/v1/accounts/{id}/budgets
3
Helix generates the agent's config and rewrites it
Helix runs preloop agents onboard hermes (or the equivalent API call) to rewrite the agent's local config to route through Preloop. The agent gets a short-lived runtime token. Its MCP servers are proxied. Its model traffic routes through the gateway.
4
Agent appears in Helix dashboard — ready to work
The agent shows up in Helix's agent dashboard with status, repo assignments, budget tracking, and policy bindings. The developer never visited Preloop.

3. Agent Makes a Risky Tool Call

1
Agent calls bash deploy production
The agent issues a tool call. It routes through Preloop's MCP proxy (the agent's config was rewritten to point there).
2
Preloop evaluates the policy
The policy says: bash on main with args containing 'deploy' → require_approval. Preloop creates an ApprovalRequest.
3
Helix receives the approval request and surfaces it
Preloop's webhook (or Helix polling Preloop's API) delivers the approval request to Helix. Helix shows it in the PR UI — a banner: "helix-reviewer wants to deploy to production. [Approve] [Deny]." It also sends a Slack notification.
4
Human approves in Helix UI → Helix tells Preloop → tool executes
The human clicks Approve. Helix calls POST /approval/{id}/decide on Preloop's API. Preloop releases the blocked tool call. The agent's bash deploy production runs. Everything is audited.

The Programmatic API Surface

Every Preloop feature has a REST endpoint. Helix's Governance Adapter calls these. Here's the complete mapping of what Helix configures and how:

Helix ActionPreloop REST CallWhat It Creates/Updates
Enroll agent POST /api/v1/agents/onboard ManagedAgent, ManagedAgentCredential, config rewrite
Install runtime plugin POST /api/v1/agents/{id}/install-plugin Agent Control WebSocket channel enabled
Assign models to agent PUT /api/v1/managed-agents/{id}/bindings ManagedAgentAIModelBinding (which models, budgets, fallback)
Set agent budget PUT /api/v1/accounts/{id}/budgets Per-agent daily/monthly spend limits
Create tool policy POST /api/v1/access-rules ToolAccessRule with CEL conditions
Update tool policy PUT /api/v1/access-rules/{id} Updated conditions, priority, action
Create approval workflow POST /api/v1/approval-workflows ApprovalWorkflow (timeout, required approvers, channels)
Set allowed models PUT /api/v1/subject-governance/{id} Subject-scoped allowed_models, model_budgets
Query agent spend GET /api/v1/api-usage?agent_id={id} Token usage, cost by model, by session
Query audit trail GET /api/v1/audit/logs?agent_id={id} All tool calls, policy decisions, approvals
Send operator command POST /api/v1/agents/{id}/control/commands Delivers message to agent via WebSocket
Respond to approval POST /approval/{id}/decide Approve or deny a pending tool call
Add API key for provider POST /api/v1/secrets Encrypted-at-rest provider credential
Register external MCP server POST /api/v1/mcp-servers External MCP server available for proxying
✅ Everything Is Programmable

There is no feature in Preloop that requires the UI. Every operation has a REST endpoint. Helix can configure every aspect of governance — agent enrollment, policy creation, budget setup, approval workflows, model bindings, secret management — through API calls. The Helix Governance Adapter is purely a REST client with translation logic.

Configuration-as-Code — The .helix/ Directory

Every repo gets a .helix/ directory. These files are version-controlled, reviewed in PRs, and automatically applied by Helix on push. Here's the complete layout:

my-repo/
├── .helix/
│   ├── policies.yaml          # Tool access rules, agent permissions, branch policies
│   ├── AGENTOWNERS             # Who is responsible for which paths (like CODEOWNERS)
│   ├── agents.yaml             # Which agents are assigned to this repo
│   ├── budgets.yaml            # Per-agent, per-PR budget limits
│   ├── prompts/                # Prompt-as-code directory
│   │   ├── review.yaml         # Prompt template for code review
│   │   ├── test.yaml           # Prompt template for test generation
│   │   └── refactor.yaml       # Prompt template for refactoring
│   ├── evaluation/             # PromptFoo evaluation tests
│   │   └── review-quality.yaml # Assertions for code review quality
│   └── models.yaml             # Allowed models + fallback chains for this repo
├── src/
├── tests/
└── README.md
💡 How It Works

On every push to the repo, Helix's webhook handler:

1. Reads .helix/policies.yaml → translates to Preloop ToolAccessRule API calls
2. Reads .helix/agents.yaml → creates/updates ManagedAgent bindings in Preloop
3. Reads .helix/budgets.yaml → sets budget enforcement rules in Preloop
4. Reads .helix/models.yaml → configures allowed models + fallback chains in Preloop
5. Reads .helix/AGENTOWNERS → configures approval routing in Preloop
6. Loads .helix/prompts/*.yaml → registers prompt templates in Helix Prompt Registry
7. Queues .helix/evaluation/*.yaml → runs PromptFoo evaluation in CI

All enforcement is live immediately. No manual steps. No Preloop UI. Git is the control plane.

Full PR Lifecycle — End to End

Here's what happens from the moment a PR is opened to when it's merged — with agents and humans working together, governed by Preloop, orchestrated by Helix.

1
Developer opens PR #142
Forgejo fires a webhook. Helix reads .helix/agents.yaml to find the assigned agent for this repo (helix-reviewer).
2
Helix dispatches agent for automated review
Helix loads the review prompt from .helix/prompts/review.yaml, injects the PR diff and context, and sends it to helix-reviewer via the Agent Control WebSocket (through Preloop). The agent begins reviewing.
Model call → Preloop Gateway → budget check ($5.00/day for this agent on this repo) → LiteLLM → DeepSeek V4 Pro
3
Agent posts review comments
The agent finds issues and wants to post comments on the PR. It calls the github MCP tool (proxied through Preloop). Preloop checks the policy: github on any branch → allow. Tool executes. Comments appear on the PR.
4
Agent wants to fix an issue — calls file_write
The agent generates a fix and calls file_write src/config.py. Preloop checks policy: file_write on feat/* → allow. Tool executes. File is written.
5
Agent wants to commit and push — triggers approval
The agent calls bash git push origin feat/pr-142. Preloop evaluates: bash containing 'push' on feat/* → require_approval. Preloop creates an ApprovalRequest and notifies Helix.
6
Human approves in Helix UI
The developer sees an approval prompt in the PR page: "helix-reviewer wants to push changes. [Approve] [Deny] [View Diff]." They click Approve. Helix calls POST /approval/{id}/decide on Preloop. The blocked git push executes.
7
CI runs, agent monitors
Gitea Actions runs the CI pipeline. If it fails, helix-tester (assigned via .helix/agents.yaml) picks up the failure log, diagnoses the issue, and posts a fix. The cycle repeats until green.
8
PR merged — cost summary posted
The PR is merged. Helix queries Preloop's GET /api/v1/api-usage?session_id={id} and posts a cost summary as a PR comment:
PR #142 Cost Summary
helix-reviewer: 3 sessions · $2.14
helix-tester: 1 session · $0.87
Total: $3.01 · Budget: $5.00 · ✅ Under budget
Audit trail: 47 tool calls · 3 approvals · 0 denials

🧬 The Final Architecture

Helix = Forgejo (git + PRs) + Agent Registry + Prompt Infrastructure + Agent-to-Agent PR Review + Marketplace + .helix/ Config-as-Code + Unified UI
Helix Governance Adapter = ~2,000 lines of Python translating .helix/*.yaml → Preloop REST API calls
Preloop = MCP Firewall + Model Gateway + Budgets + Approvals + Audit + Secret Custody + Agent Control WS

The developer sees Helix. The enforcement runs on Preloop. The boundary is a REST API. The repo is the source of truth.

💡 Why This Works

Preloop's architecture was designed for this. It already separates its REST API from its frontend (Lit + Shoelace). The frontend is optional — every operation is available via the API. Helix simply replaces Preloop's frontend with its own, richer, git-native one. Preloop's backend doesn't know or care which frontend is calling it. This is the cleanest possible integration: Helix as the presentation layer, Preloop as the governance engine, separated by a well-defined API contract.
